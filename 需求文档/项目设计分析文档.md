## 1. 项目定位

劳动合同法律助手是一个基于大模型与 RAG 的劳动合同智能咨询系统，面向劳动合同高频咨询、法规依据查询、维权流程指引和合同条款风险提示等场景。

项目建议以“劳动合同智能问答 + 法律依据引用 + 维权流程指引”为 MVP 核心，首版不训练专有大模型，而是通过现有大模型能力、法规政策知识库和检索增强生成流程实现可追溯问答。

系统定位为劳动合同法律信息检索与咨询辅助工具，不替代律师、仲裁机构或法院意见。对于重大争议、诉讼策略、金额较大或证据复杂的问题，系统应提示用户咨询专业律师或当地劳动保障部门。

## 2. 建设目标

### 2.1 产品目标

| 目标类别 | 设计目标 | 衡量方式 |
| --- | --- | --- |
| 问答有效性 | 用户输入劳动合同相关问题后，系统能给出清晰、结构化、可追溯的回答。 | 常见问题命中率不低于 85%，回答包含依据、流程、注意事项。 |
| 法律依据可信 | 每条关键结论能关联法规、司法解释或政策文件。 | 引用来源覆盖率不低于 90%，来源包含名称、条款或文档标题。 |
| 操作可落地 | 不只回答“能不能”，还给出下一步维权流程和证据准备建议。 | 80% 以上测试问题给出可执行步骤。 |
| 实训可实现 | 控制复杂度，不训练模型，优先使用现有大模型 API 和向量库。 | MVP 可在实训周期内完成部署演示。 |
| 用户体验 | 界面聚焦咨询，不做复杂法律门户。 | 用户能在 3 步内完成提问、查看依据、保存结果。 |

### 2.2 项目管理目标

结合项目计划表，整体建设目标是形成可运营、可审计、可持续迭代的 AI RAG 知识库系统，支持知识源接入、文档解析与清洗、Embedding 与向量库、RAG 检索编排、LLM 问答服务、Web 管理、用户权限、日志审计、评测体系、试运行与上线。

项目采用“阶段门 + 敏捷迭代”的混合模式：

1. 使用 PMP 方法管控范围、进度、质量、风险和变更。
2. 研发阶段采用 2 周迭代，先以 MVP 验证知识接入、检索质量和权限安全。
3. 后续逐步扩展运营能力、评测能力和治理能力。

| 场景 | 用户问题示例 | 系统应答重点 |
| --- | --- | --- |
| 加班工资 | 公司要求加班但是不给钱怎么办？ | 加班工资规则、证据保留、协商/投诉/仲裁路径。 |

| 合同审查 | 竞业限制条款合理吗？ | 高风险条款识别、风险解释、修改建议。 |

## 4. 项目范围与边界

### 4.1 MVP 范围

1. 劳动合同相关智能问答：支持自然语言提问，围绕劳动合同签订、履行、变更、解除、终止、经济补偿、加班工资、试用期、社保等主题回答。
2. 法律依据展示：回答中展示依据来源、文档名称、条款或政策标题，并支持查看原文片段。
3. 维权流程指引：针对争议类问题，输出协商、投诉举报、劳动仲裁、诉讼等路径，并提示材料准备。
4. 注意事项提示：提示证据留存、时效、地方差异、复杂案件咨询专业律师等限制。
5. 知识库管理：管理员可上传法规政策文件，进行解析、切分、向量化、启停用和版本管理。
6. 问答反馈：用户可对答案点赞、点踩，提交错误反馈，供管理员优化知识库和提示词。

### 4.2 暂不纳入范围

1. 不提供律师代理、法律文书代写或付费法律服务。
2. 不承诺覆盖所有地方性劳动政策，首版以全国性法规政策和示例材料为主。
3. 不训练专有大模型，首版采用大模型 API 或本地开源模型调用加 RAG 检索。
4. 不处理用户上传的大量敏感个人材料作为训练数据，仅支持会话级解析与必要脱敏。
5. 不替代正式法律、医疗、财务等高风险专业审查。
6. 不包含大规模业务系统重构、非授权数据采集或未纳入合同的多语言全量治理。

## 5. 总体设计方案

### 5.1 能力分层

系统可按以下能力分层设计：

| 层级 | 主要能力 |
| --- | --- |
| 用户交互层 | 提问、查看回答、查看引用、继续追问、提交反馈、查看历史、导出结果。 |
| 智能咨询层 | 意图识别、问题分类、范围判断、结构化回答、安全拒答、合同条款风险提示。 |
| RAG 编排层 | 检索策略选择、关键词检索、向量检索、结果重排、提示词组装、引用溯源。 |
| 知识处理层 | 文档上传、解析、清洗、切分、Embedding、索引写入、版本维护。 |
| 数据治理层 | 元数据管理、知识源状态、权限边界、反馈闭环、日志审计、质量评测。 |
| 运营管理层 | 知识更新、FAQ 维护、低质量答案复盘、评测集维护、运营监控。 |

### 5.2 核心业务链路

```text
用户提问
  ↓
问题范围识别与意图分类
  ↓
检索法规、司法解释、人社政策、FAQ、示例合同等知识片段
  ↓
按相关度、时效性和来源权威性排序
  ↓
大模型基于检索结果生成结构化回答
  ↓
展示结论摘要、法律依据、维权流程、证据清单、注意事项
  ↓
用户继续追问、保存结果或提交反馈
  ↓
管理员根据反馈更新知识库、FAQ 和提示词
```

### 5.3 知识入库链路

```text
管理员上传知识文档
  ↓
记录来源、标题、发布日期、适用范围和版本号
  ↓
解析文档文本
  ↓
按条款、章节或自然段切分
  ↓
保留文档名、章节、条款号、来源 URL、入库时间等元数据
  ↓
生成向量并建立检索索引
  ↓
管理员审核启用
  ↓
参与正式问答检索
```

## 6. 功能模块分析

### 6.1 智能咨询模块

用户可输入劳动合同相关自然语言问题，系统根据问题类型自动选择回答策略。首版问题类型包括：

1. 合同签订
2. 试用期
3. 工资支付
4. 加班工资
5. 休息休假
6. 社保
7. 解除或终止劳动合同
8. 经济补偿
9. 劳动仲裁
10. 证据准备

核心设计要求：

| 需求编号 | 设计内容 | 优先级 | 验收标准 |
| --- | --- | --- | --- |
| FR-001 | 支持用户输入劳动合同相关自然语言问题。 | P0 | 输入完整问题后，系统能返回结构化答案。 |
| FR-002 | 系统识别问题意图并展示问题分类。 | P1 | 至少覆盖 10 类高频劳动合同场景。 |
| FR-003 | 回答结构固定为结论摘要、法律依据、维权流程、注意事项。 | P0 | 每个字段均有内容；超出范围时给出拒答说明。 |
| FR-004 | 支持连续追问，保留当前会话上下文。 | P1 | 追问证据准备时能承接上一问题。 |
| FR-005 | 支持答案复制、收藏或导出为文本。 | P2 | 用户可保存咨询结果。 |

### 6.2 法律依据与引用溯源模块

系统生成答案时必须优先引用知识库检索结果，避免凭空编造法律结论。依据展示不要求长篇原文复制，重点展示来源名称、条款/章节、摘要和相关片段。

| 需求编号 | 设计内容 | 优先级 | 验收标准 |
| --- | --- | --- | --- |
| FR-006 | 每条核心法律结论展示至少一个来源引用。 | P0 | 答案页能看到来源列表。 |
| FR-007 | 支持展开查看依据片段。 | P0 | 点击来源后显示检索命中的原文摘要。 |
| FR-008 | 引用来源按权威性和相关度排序。 | P1 | 法规优先于普通解释性材料。 |
| FR-009 | 当知识库依据不足时，系统提示“依据不足”，不得强行给出确定性结论。 | P0 | 超范围或依据不足问题能安全拒答。 |

引用来源建议展示字段：

| 字段 | 说明 |
| --- | --- |
| 来源名称 | 法规、司法解释、政策文件、FAQ 或示例合同标题。 |
| 条款/章节 | 对应条文、章节、政策标题或 FAQ 编号。 |
| 来源类型 | 法律法规、司法解释、人社政策、地方政策、FAQ、示例合同。 |
| 发布日期 | 文档发布时间或版本日期。 |
| 适用范围 | 全国性、地方性或特定场景。 |
| 原文摘要 | 检索命中的原文片段。 |
| 来源链接 | 可追溯的权威来源地址。 |

### 6.3 维权流程与证据清单模块

对于劳动争议类问题，系统应将回答从“解释规则”扩展为“下一步怎么做”。

流程建议覆盖：

1. 与用人单位沟通。
2. 保存劳动关系、工资、考勤、加班、解除通知等证据。
3. 向劳动保障监察部门投诉或举报。
4. 申请劳动仲裁。
5. 必要时提起诉讼。

证据清单建议覆盖：

| 证据类型 | 示例 |
| --- | --- |
| 劳动关系证据 | 劳动合同、入职通知、工牌、工作邮件、工作群记录。 |
| 工资证据 | 工资流水、工资条、转账记录、个税记录。 |
| 工时考勤证据 | 打卡记录、排班表、加班通知、审批记录。 |
| 沟通证据 | 微信聊天、短信、邮件、录音说明。 |
| 工作事实证据 | 工作成果、客户沟通记录、系统账号、绩效记录。 |
| 社保记录 | 社保缴纳证明、公积金缴存记录。 |

### 6.4 合同条款审查模块

合同审查作为 P1 功能，用户可粘贴劳动合同条款或上传示例文本，系统识别明显风险点并给出修改建议。首版不做完整法律审查，只提供辅助提示。

| 审查类型 | 识别内容 | 输出内容 |
| --- | --- | --- |
| 试用期 | 试用期过长、试用期工资过低、重复约定试用期。 | 风险等级、依据、建议表述。 |
| 工资工时 | 工资结构不清、加班费约定模糊、休息休假缺失。 | 风险说明、需补充字段。 |
| 解除终止 | 单方解除条件过宽、违约金约定不合理。 | 风险解释、谈判建议。 |
| 保密竞业 | 竞业限制对象、期限、补偿约定不清。 | 风险点、补充建议。 |

### 6.5 知识库管理模块

| 需求编号 | 设计内容 | 优先级 | 验收标准 |
| --- | --- | --- | --- |
| FR-010 | 管理员上传法规、司法解释、政策文件、FAQ 文档。 | P0 | 支持 PDF、DOCX、TXT 至少一种格式入库。 |
| FR-011 | 系统自动解析、切分、生成向量索引。 | P0 | 上传文件后可被问答检索命中。 |

### 6.6 问答反馈模块

用户可对回答进行点赞、点踩或提交错误反馈。反馈应追踪到具体问题、答案、引用片段和处理状态，用于管理员优化知识库、提示词和高频问答。

## 7. 数据与知识库分析

知识库是系统可信度的基础。首版建议构建“小而准”的资料库，不追求大而全。

### 7.1 数据来源优先级

| 数据类型 | 示例 | 用途 | 维护要求 |
| --- | --- | --- | --- |
| 法律法规 | 劳动合同法、劳动法等。 | 提供基础法律依据。 | 保留条款结构、发布日期、版本。 |
| 司法解释 | 劳动争议相关司法解释。 | 解释争议处理口径。 | 标注文号、发布时间、适用范围。 |
| 人社政策 | 劳动合同、工资支付、电子劳动合同等政策文件。 | 补充实务操作说明。 | 记录来源链接与有效性。 |
| 地方政策 | 地方工资支付、劳动监察、仲裁服务流程。 | 补充地方差异。 | 标注地区、有效期、适用范围。 |
| FAQ | 加班费、未签合同、试用期等高频问题。 | 提升回答稳定性。 | 由管理员审核后发布。 |
| 示例合同 | 劳动合同样本、风险条款样例。 | 支持合同审查演示。 | 使用脱敏样本，避免真实个人信息。 |

### 7.2 数据处理规则

1. 文档入库前记录来源、标题、发布日期、适用范围和版本号。
2. 切分粒度建议以条款、章节或自然段为单位，避免过长片段影响检索。
3. 每个片段保留元数据：文档名、章节、条款号、来源 URL、入库时间。
4. 过期或不确定有效性的文件标记为“待确认”，默认不参与正式回答。
5. 用户个人材料仅用于当前会话分析，不进入公共知识库。

## 8. RAG 设计分析

### 8.1 检索策略

RAG 流程建议采用混合检索：

1. 关键词检索：适合匹配法规名称、条款号、政策关键词。
2. 向量检索：适合自然语言问题与知识片段的语义匹配。
3. 元数据过滤：过滤停用、过期、待确认、非适用范围文档。
4. 结果重排：综合相关度、权威性、时效性和来源类型排序。
5. 引用绑定：将最终回答中的关键结论绑定到检索片段。

### 8.2 回答生成约束

标准回答结构：

1. 结论摘要：先给出简明判断。
2. 法律依据：列出相关法规、司法解释或政策文件来源。
3. 维权流程：按协商、投诉、仲裁、诉讼等路径说明。
4. 证据清单：提示合同、工资、考勤、聊天记录等材料。
5. 注意事项：提示时效、地方差异、复杂案件咨询专业律师。

模型回答必须遵守：

1. 不编造不存在的法律条款、文号和流程。
2. 无依据时说明依据不足并安全拒答。
3. 不承诺仲裁或诉讼一定胜诉。
4. 不指导伪造证据、规避法律或恶意维权。
5. 对复杂个案提示专业咨询。

## 9. 权限与管理分析

| 角色 | 主要权限 | 限制 |
| --- | --- | --- |
| 游客 / 普通用户 | 提问、查看答案、查看引用、提交反馈。 | 不能上传知识库文件，不能查看他人会话。 |
| 登录用户 | 保存问答历史、导出结果、继续追问。 | 只能管理自己的记录。 |
| 管理员 | 上传/停用知识文档、查看反馈、维护 FAQ、配置提示词。 | 不应查看用户敏感原文，除非获得授权。 |
| 项目演示评审 | 查看演示账号、测试问答、查看系统说明。 | 只读访问，不修改知识库。 |

权限设计重点：

1. 管理能力必须受登录和角色限制。
2. 用户只能访问自己的问答历史。
3. 知识库文档必须有启用、停用、待确认等状态。
4. 管理员操作应记录日志，便于审计和追溯。
5. 涉及敏感个人信息时应做提示、脱敏和访问限制。

## 10. 安全与合规分析

### 10.1 主要风险

| 风险 | 影响 | 应对策略 |
| --- | --- | --- |
| 法律依据过期 | 回答错误或误导用户。 | 保留来源和版本，管理员定期更新，答案提示发布日期。 |
| 大模型幻觉 | 编造条款或流程。 | 强制基于检索片段回答，无依据时拒答。 |
| 问题超出劳动合同范围 | 回答质量下降。 | 设置范围识别和安全拒答，引导用户咨询专业机构。 |
| 用户输入隐私信息 | 存在隐私泄露风险。 | 前端提示脱敏，日志脱敏，禁止进入公共知识库。 |
| 开发周期有限 | 功能无法全部完成。 | 优先 P0 需求，P1/P2 作为扩展。 |
| 权限泄露 | 无权限用户访问敏感知识或管理能力。 | 权限隔离、安全测试和审计日志。 |
| 成本超支 | 模型调用、向量库和算力成本不可控。 | 成本看板、限流策略、预算基准和采购评估。 |

### 10.2 免责声明

系统应在首页、答案页或导出结果中明确提示：

```text
本系统仅提供劳动合同法律信息检索与咨询辅助，不构成正式法律意见，不替代律师、仲裁机构或法院意见。
```

## 18. 设计结论

本项目应以可追溯、可解释、可运营为核心设计原则，优先完成劳动合同智能问答、法律依据引用、维权流程指引和知识库管理闭环。RAG 设计应坚持“先检索、后生成、有依据才回答”的原则，避免大模型幻觉和法律误导。

从项目实施角度看，26 周建设周期较适合采用阶段门管控与敏捷迭代结合的方式。前期应重点完成需求基线、知识源清单、数据处理规则、权限模型和质量指标定义；MVP 阶段优先验证文档入库、检索命中、引用溯源和结构化回答；集成测试和试运行阶段重点关注权限安全、响应时延、引用完整性、反馈闭环和运营移交。
