# G:\LaborLawAI 项目风险与问题分析

审查日期：2026-07-21  
审查范围：仓库结构、Git 跟踪状态、后端 Spring Boot/MyBatis-Plus 代码、前端 Vue/Vite 代码、数据库脚本、文档一致性、基础测试/构建/依赖审计。  
说明：本报告只记录问题和风险，不包含真实密码、Token、私钥等敏感原文。

## 一、执行摘要

本次复核发现：项目已经具备前端知识库管理页面、后端知识库文档 CRUD、配置型管理员 RBAC、审计日志和部分测试；但当前仓库仍存在若干会影响安全、可维护性和交付可信度的问题。

最高优先级问题集中在四类：

1. **仓库仍跟踪大量生成物和本地运行文件**：`frontend/node_modules`、`backend/target`、`frontend/dist`、`.npm-cache`、`.npm-global`、`.idea`、`.lark-auth` 等仍在 Git 索引中；`.gitignore` 已忽略但未从索引移除。本次运行验证后，`backend/target` 与 `frontend/dist` 立即产生 Git 变更，证明该风险仍存在。
2. **敏感配置/本地授权材料治理不足**：前端本地环境文件、后端本地配置文件、飞书/Lark 授权日志和二维码文件被跟踪。即使当前多为本地占位值或已打码内容，也会给后续误提交真实凭据留下通道。
3. **后端测试当前不通过**：`mvn test` 退出码为 1。控制器测试切片加载时扫描到 MyBatis Mapper，但测试上下文没有 `sqlSessionFactory/sqlSessionTemplate`，导致多个控制器测试无法加载 ApplicationContext。
4. **公共接口可能暴露问答历史/引用内容**：`/api/qa/records`、`/api/qa/citations`、`/api/knowledge/**` 不在 `/api/admin/**` 鉴权拦截范围内。如果这些数据包含用户问题、问答摘要、引用片段或内部知识库状态，在公网环境会形成隐私和数据泄露风险。

## 三、P0 高风险问题

### P0-1：生成物、依赖目录和本地工具仍被 Git 跟踪

**证据**

- `git ls-files` 仍列出 `frontend/node_modules/**`、`backend/target/**`、`frontend/dist/**`、`.npm-cache/**`、`.npm-global/**` 等。
- 本次运行 `mvn test` 与 `npm run build` 后，`backend/target` 与 `frontend/dist` 立即变成 modified，说明普通验证动作会污染工作区。
- 现有 `项目风险问题分析.md` 曾写明 `backend/target`、`frontend/dist` 已从 Git 跟踪中移除，但当前仓库实际状态与该结论不一致。

**风险影响**

- 仓库体积膨胀，拉取/提交/差异审查成本升高。
- Windows/macOS/Linux 原生依赖混入仓库，跨平台可重复构建能力下降。
- 构建或测试会修改被跟踪产物，导致开发者误提交二进制、缓存和测试报告。
- 安全审查和代码审查被大量第三方文件干扰，真实源码变更更难定位。

**建议处理**

1. 使用 `git rm -r --cached` 从索引移除：`frontend/node_modules`、`backend/target`、`frontend/dist`、`.npm-cache`、`.npm-global`、`.idea`、`.vscode`、`.lark-auth`。
2. 保留 `.gitignore` 当前规则，并补充 `.lark-auth/`、`.npm-cache/`、`.npm-global/` 等本地工具目录规则。
3. 提交一次“停止跟踪生成物和本地文件”的专门提交。
4. 后续通过 `package-lock.json`、`pom.xml` 和 README 安装步骤恢复依赖，而不是提交依赖目录。

### P0-2：本地环境文件和授权材料被跟踪

**证据**

- Git 跟踪了 `frontend/.env`、`frontend/.env.development`。
- Git 跟踪了 `backend/src/main/resources/application-local.yml`。
- Git 跟踪了 `.lark-auth/config-init.*` 与授权二维码图片。
- 后端主配置和本地配置仍包含本地数据库默认口令占位；完整数据库脚本还包含默认管理员账号初始化逻辑。

**风险影响**

- 本地 Token、数据库口令、第三方授权材料一旦替换为真实值，极易被误提交。
- Vite 环境变量会在构建期写入前端产物，任何 `VITE_*` 管理员 Token 都不应被视为服务端秘密。
- 飞书/Lark 授权二维码、用户码、日志即使短期有效，也不应进入代码仓库。

**建议处理**

1. 仅保留 `frontend/.env.example`、`application-local.example.yml` 等模板文件。
2. 从 Git 索引移除真实/本地环境文件和 `.lark-auth`，并确认 `.gitignore` 覆盖这些路径。
3. 生产部署中不要把管理员 Token 放入前端静态包；改为登录态、短期访问令牌或服务端会话。
4. 默认管理员账号应改为部署时初始化，首次启动强制改密或通过环境变量创建，避免脚本中固定弱口令。

### P0-3：公共接口可能暴露问答历史、引用快照和知识库元数据

**证据**

- `WebMvcConfig` 只对 `/api/admin/**` 注册管理员访问拦截器。
- `QaRecordController` 暴露 `/api/qa/records`。
- `QaCitationController` 暴露 `/api/qa/citations`。
- `KnowledgeSourceController` 暴露 `/api/knowledge/documents` 和 `/api/knowledge/chunk-refs`。
- `QaRecordVO` 包含问题、结论摘要、行动步骤、证据清单、注意事项等字段；`QaCitationVO` 包含引用快照、来源 URL、引用片段等字段。

**风险影响**

- 如果问答历史包含用户劳动争议细节、身份线索、企业名称或案件事实，公网访问会形成隐私泄露。
- 引用片段和知识库状态可能暴露内部数据治理进度、未发布来源或低质量片段。
- 法律咨询系统对来源可信度和用户隐私敏感，公开历史数据会降低系统可信度。

**建议处理**

1. 明确这些接口是否是“公开前台检索”还是“管理员/内部审计”。
2. 若属于内部数据，纳入 `/api/admin/**` 或引入用户登录鉴权与按用户/角色隔离。
3. 若必须公开，返回字段应最小化：只展示已发布、已审核、脱敏后的公开来源，默认不返回用户问答历史。
4. 对问答入库前增加 PII 检测、脱敏和数据保留策略。

### P0-4：后端测试失败，当前不能证明后端功能可回归

**证据**

- `mvn test` 当前失败。
- 失败集中在控制器测试启动 ApplicationContext：`kbChunkRefMapper` 需要 `sqlSessionFactory/sqlSessionTemplate`。
- 失败与测试切片和 Mapper 扫描边界有关，不是单个业务断言失败。

**风险影响**

- README 和历史风险文档中“后端测试通过”的结论已经过期。
- 后续修改鉴权、异常处理或接口时缺少可靠回归网。
- CI 若接入后会直接失败；若不接入 CI，则失败可能长期被忽略。

**建议处理**

1. 将 Mapper 扫描从主应用类迁移到单独配置，并避免 `@WebMvcTest` 自动加载 MyBatis Mapper。
2. 或在控制器切片测试中显式排除 MapperScan/MyBatis 自动配置，并只 Mock 控制器依赖的 Service。
3. 用 `spring-boot-testcontainers` 或 H2/MySQL Testcontainer 补充真正的数据访问集成测试。
4. 修复后重新运行 `mvn test`，并把测试命令加入 CI。

## 四、P1 重要问题

### P1-1：数据库脚本存在两套且结构不一致

**证据**

- `backend/src/main/resources/db/init.sql` 只有 `kb_document`、`kb_chunk`、`qa_session`、`qa_feedback` 等旧结构。
- 当前实体使用 `public_id`、`document_type`、`canonical_source_url`、`kb_chunk_ref`、`qa_answer`、`qa_answer_citation` 等字段/表。
- `backend/db/init.sql` 是完整结构，但包含大量 `DROP TABLE`，且不在 `src/main/resources` 当前同名脚本中体现。

**风险影响**

- 新成员或部署脚本若使用资源目录下的 `init.sql`，会创建与代码不兼容的数据库，运行时出现缺表/缺列错误。
- 完整脚本若误用于已有环境，会因 `DROP TABLE` 造成数据丢失。
- 文档、启动脚本、应用资源脚本不一致，会导致“本地能跑、别人跑不起来”。

**建议处理**

1. 明确唯一权威 schema 来源：推荐引入 Flyway 或 Liquibase 迁移脚本。
2. 删除或改名旧版 `src/main/resources/db/init.sql`，避免误用。
3. 把破坏性初始化脚本和增量迁移脚本分离，生产环境禁止自动执行 `DROP TABLE`。
4. README 明确开发库初始化和生产库迁移流程。

### P1-2：管理员 RBAC 仍停留在环境变量 Token 模式

**证据**

- `AdminAccountRegistry` 从环境变量解析 `username|token|roles`。
- Token 以明文配置值形式存在于环境变量和前端构建变量中；没有过期、吊销、轮换、哈希存储和登录审计表。
- 审计日志目前写入应用日志，不持久化到数据库审计表。

**风险影响**

- Token 泄露后只能整体换配置并重启，无法按账号吊销。
- 前端构建包中的管理员 Token 无法保密。
- 无法满足正式系统对账号生命周期、最小权限、可追溯审计和离职回收的要求。

**建议处理**

1. 使用数据库账号、角色、权限表替代环境变量 Token。
2. 引入 Spring Security 或等价安全框架。
3. 使用短期访问令牌、刷新令牌或服务端会话，并支持吊销和过期。
4. 审计日志持久化到 `sys_audit_log`，并记录 actor、动作、目标、结果、请求来源和时间。

### P1-3：输入校验不足，知识库元数据可信度可能下降

**证据**

- 创建/更新 DTO 只校验 `title` 和 `document_type` 非空。
- `status`、`document_type`、`jurisdiction_code`、`canonical_source_url`、`authority_level` 等缺少枚举、格式、长度和范围校验。
- 服务层会把空状态默认成 `DRAFT`，但不会拒绝未知状态。

**风险影响**

- 非法状态、错误辖区、非权威 URL 或异常权威等级可进入数据库。
- 前端筛选和法律依据可信度排序可能被脏数据破坏。
- 后续 RAG 检索会依赖这些元数据，错误元数据会影响回答依据质量。

**建议处理**

1. 为 `document_type`、`status`、`jurisdiction_code` 建立白名单枚举。
2. URL 使用格式校验和来源域名策略；必要时标记“官方/非官方来源”。
3. `authority_level` 增加范围约束，例如 1-999。
4. 数据库层增加 CHECK 约束或参考表，避免绕过 API 写入脏数据。

### P1-4：删除与版本/索引生命周期未形成闭环

**证据**

- `KnowledgeDocumentService.deleteById` 直接按主键删除文档。
- 完整数据库模型包含文档版本、入库任务、分片引用、Elasticsearch 文档 ID 等生命周期字段，但当前 CRUD 只管理文档元数据。
- 删除失败时会抛出 `IllegalStateException`，全局异常最终返回 500，客户端难以区分“存在关联数据不能删”和系统错误。

**风险影响**

- 文档删除后可能遗留向量索引、分片、引用快照或外部文件对象。
- 直接硬删可能破坏问答引用的可追溯性。
- 并发编辑/删除没有使用 `lock_version`，存在覆盖更新和状态竞争。

**建议处理**

1. 优先采用软删除/禁用状态，不直接硬删已发布或已引用文档。
2. 删除前检查版本、分片、引用、索引任务和问答引用。
3. 删除知识文档时同步创建索引清理任务。
4. 更新接口引入 `lock_version` 乐观锁，状态变更使用明确状态机。

### P1-5：缺少 CI、自动测试门禁和持续安全扫描

**证据**

- `.github/workflows` 不存在。
- 当前需要人工运行 `mvn test`、`npm test`、`npm audit`、`npm run build`。
- Maven 依赖尚未看到 OWASP Dependency-Check、Maven Enforcer 或类似供应链扫描配置。

**风险影响**

- 后端测试失败不会在提交前自动暴露。
- 依赖漏洞、构建失败、格式问题容易进入主分支。
- 生成物被跟踪的问题也缺少自动阻断。

**建议处理**

1. 新增 CI：后端 `mvn test`，前端 `npm ci && npm test && npm run build && npm audit`。
2. 增加禁止提交生成物/大文件/本地环境文件的检查。
3. Maven 增加依赖版本收敛和漏洞扫描。
4. 对 PR 输出测试结果和构建产物体积摘要。

## 五、P2 中低风险和体验问题

### P2-1：前端体积仍偏大

**证据**

- `npm run build` 成功，但 CSS 约 363.75 kB，主业务 JS 约 458.48 kB。
- Element Plus 和相关组件一次性引入较多样式与模块。

**风险影响**

- 弱网或移动端首屏加载慢。
- 后续功能增加后更容易超过性能预算。

**建议处理**

1. Element Plus 按需引入组件和样式。
2. 路由级懒加载，把管理页、历史页和前台咨询拆分 chunk。
3. 建立 gzip/brotli 后的体积预算。

### P2-2：前端仍有较多演示/占位功能

**证据**

- `App.vue` 中智能咨询、合同审查、快捷问题和依据卡片主要为静态演示内容。
- README 也说明完整 RAG 入库链路、正式登录中心、依据引用和问答历史仍需补充。

**风险影响**

- 用户容易误以为系统已经具备完整 AI 咨询能力。
- 演示法律结论若长期不更新，可能与最新法律、地方政策或个案事实不匹配。

**建议处理**

1. 在 UI 上明确标注“演示数据/未连接真实模型”。
2. 真正接入 AI 前，避免把静态示例呈现为实时法律建议。
3. 对所有回答加入依据版本、发布日期、效力状态和免责声明。

### P2-3：构建存在非阻断警告

**证据**

- Vite/Rollup 构建成功，但提示 `@vueuse/core` 中 PURE 注释位置无法解释，将移除注释。

**风险影响**

- 当前不阻断发布，但会掩盖未来真正重要的构建警告。

**建议处理**

1. 评估 Vite、Element Plus、VueUse 版本组合。
2. 升级或锁定可消除警告的依赖版本。
3. CI 中将新增警告趋势纳入观察，而不是立即把所有 warning 设为失败。

## 六、建议处理顺序

1. **先治理仓库索引**：移除 `node_modules`、`target`、`dist`、本地缓存、IDE 文件、Lark 授权文件和本地 `.env`。
2. **修复后端测试失败**：恢复 `mvn test` 作为可信回归基线。
3. **收口敏感接口**：确认 `/api/qa/**`、`/api/knowledge/**` 的公开边界，必要时纳入鉴权。
4. **统一数据库迁移方案**：以 Flyway/Liquibase 或唯一 SQL 入口替代两套不一致脚本。
5. **升级管理员安全模型**：从前端内置 Token 与环境变量 RBAC 迁移到正式账号体系。
6. **建立 CI 与供应链扫描**：把测试、构建、审计和禁止提交生成物变成自动门禁。
7. **再处理前端体积和演示体验**：按需加载、路由拆分、清晰标识演示内容。
