# G:\AI-law-creater 项目风险与问题分析

## 总体判断

这个项目已经具备了后端基础、前端基础、统一响应对象和部分知识库文档接口，但仍然存在数据库初始化、权限控制、错误处理、测试覆盖、前端业务完整性等方面的风险。

## 1. 数据库初始化依赖手工操作

### 问题
`backend/db/init.sql` 内容完整，但数据库初始化仍依赖人工执行。此前 `kb_document` 表缺失时，新增接口直接返回 500，说明从零启动时的可用性不足。

### 风险
- 新环境启动失败
- 不同开发者数据库状态不一致
- CI/CD 或部署容易遗漏建表

### 建议
- 引入 Flyway 或 Liquibase
- 拆分可重复执行的 migration
- 在 README 中明确初始化顺序

## 2. 后端错误处理不够统一

### 问题
统一返回对象 `ApiResponse` 已存在，但并非所有异常都通过统一格式返回。参数校验、JSON 解析、SQL 异常等场景仍可能出现 Spring Boot 默认错误响应。

### 风险
- 前端需要兼容多种错误格式
- 调试体验不一致
- 接口契约不稳定

### 建议
- 增加全局异常处理器
- 统一处理校验错误、解析错误、数据访问错误和运行时错误
- 普通 JSON 接口尽量都返回 `ApiResponse`

## 3. 管理员接口缺少鉴权

### 问题
知识库文档接口挂在 `/api/admin/**`，但仓库中没有真正的登录鉴权或角色权限控制。

### 风险
- 任意人都可能新增、修改、删除文档
- 一旦接入公网，风险很高

### 建议
- 增加基础鉴权机制
- 对 `/api/admin/**` 做权限拦截
- 删除/更新操作增加审计日志

## 4. 配置中存在明文敏感信息

### 问题
部分本地配置和文档中存在数据库连接信息和密码示例。

### 风险
- 误提交时泄露敏感信息
- 环境混用导致问题

### 建议
- 敏感配置改用环境变量
- 本地专用配置放入未提交文件
- 如有真实密码，尽快轮换

## 5. 前端仍有占位内容

### 问题
前端部分页面仍是静态示例或 disabled 控件，知识库页面尚未形成完整 CRUD 体验。

### 风险
- 用户误以为功能已完成
- 后端接口无法在前端形成完整闭环

### 建议
- 优先完成知识库文档页真实 CRUD
- 去掉无意义的禁用控件
- 明确区分演示页和真实功能页

## 6. 前端 API 契约有潜在风险

### 问题
`frontend/src/api/http.js` 会自动解包统一响应对象，只返回 `data`。

### 风险
- 调用方拿不到 `code` 和 `message`
- 某些场景不方便做完整错误处理

### 建议
- 明确 API 层约定
- 必要时区分 `requestRaw` 和 `requestData`
- 把约定写到前端 README 中

## 7. 知识库模块仍停留在元数据层

### 问题
当前已实现文档元数据管理，但还没有进入真正的 RAG 生命周期：
- 文件上传
- 文档解析
- 切分
- 向量化
- 检索索引
- 版本管理联动

### 风险
- 模块看起来完整，但业务闭环不完整
- 后续扩展边界不清晰

### 建议
- 按阶段推进：元数据管理、文件上传、解析切分、索引构建、检索、版本管理
- 每阶段定义清晰验收标准

## 8. 测试覆盖还不够完整

### 问题
后端当前以 controller slice 测试为主，能验证接口层，但不能完全证明真实数据库链路可用。

### 风险
- 单元测试通过，集成问题仍然存在
- 字段映射和 SQL 变更容易引入回归

### 建议
- 保留 controller 测试
- 再补 service 集成测试或真实数据库测试
- 对知识库模块补至少一个真实插入/查询测试

## 9. 前端测试体系还不完整

### 问题
前端当前没有完整的测试脚本和测试框架配置。

### 风险
- 页面改动后缺少自动化回归验证
- 组件结构容易被无意改坏

### 建议
- 最小化接入 Vitest、@vue/test-utils、jsdom
- 先补知识库文档页基础渲染测试

## 10. 文档和代码容易漂移

### 问题
仓库中存在很多 README、接口文档、课程材料、建表脚本、curl 示例，容易与代码不同步。

### 风险
- 新人按文档操作失败
- 接口示例与真实实现不一致

### 建议
- 把接口文档当成代码的一部分维护
- 每次接口变更同步更新文档
- 示例请求和响应尽量使用真实返回

## 11. 项目目录边界不够清晰

### 问题
仓库中混有源码、文档、课程材料、临时脚本和构建产物。

### 风险
- 新成员不清楚哪个目录是正式项目
- 容易误提交构建产物或临时文件

### 建议
- 明确目录职责
- 临时脚本统一放到忽略目录
- 清理或忽略不必要的构建产物

## 12. 测试里存在弃用警告

### 问题
测试中有 `@MockBean` 的弃用提示。

### 风险
- 未来升级 Spring Boot 时可能需要批量调整

### 建议
- 升级规划中提前替换推荐写法
- 不要等到大版本升级时集中处理

## 优先级建议

### P0
1. 数据库初始化自动化
2. 管理员接口鉴权
3. 统一异常处理
4. 前端知识库页真实化

### P1
5. 前端测试接入
6. 后端集成测试
7. 配置敏感信息治理

### P2
8. 文档与代码同步机制
9. 构建产物和临时文件整理
10. 测试弃用警告治理

## 简短结论

当前项目的主要风险不在于功能数量，而在于：
- 环境依赖偏手工
- 安全控制偏弱
- 前后端文档与实现存在漂移风险
- 知识库模块仍停留在元数据层
- 自动化测试和部署自动化还不够
