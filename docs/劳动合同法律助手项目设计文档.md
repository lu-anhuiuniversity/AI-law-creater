## 1. 项目概述

劳动合同法律助手是一个基于大模型与 RAG（检索增强生成）的劳动合同智能咨询系统。系统面向普通劳动者、应届毕业生、实习生、企业 HR、小微企业负责人和课程项目评审者，提供劳动合同相关的智能问答、法律依据引用、维权流程指引、合同条款风险提示和知识库管理能力。

系统定位为劳动合同法律信息检索与咨询辅助工具，不替代律师、仲裁机构或法院意见。对于重大争议、诉讼策略、金额较大或证据复杂的问题，系统应提示用户咨询专业律师或当地劳动保障部门。

## 2. 设计目标

### 2.1 产品目标

1. 用户输入劳动合同相关问题后，系统返回结构化、可追溯的回答。
2. 每条关键结论尽量关联法规、司法解释、人社政策或知识库文档来源。
3. 争议类问题不仅回答规则，还提供可执行的维权流程和证据清单。
4. 不训练专有大模型，优先使用现有大模型 API 与本地或云端向量检索能力。
5. 前端界面聚焦咨询，用户在 3 步内完成提问、查看依据、保存或反馈结果。

### 2.2 技术目标

1. 基于当前项目结构继续演进，后端使用 Java 17 + Spring Boot，前端使用 Vue 3 + Vite。
2. 后端按 `controller / service / mapper / entity / dto / vo / config` 分层。
3. 知识库数据支持文档上传、解析、切分、向量化、检索、启停用和版本管理。
4. 回答生成强制基于检索片段，依据不足时安全拒答，降低大模型幻觉风险。
5. 系统保留健康检查能力，便于开发、部署和演示验证。

## 3. 范围与边界

### 3.1 MVP 范围

| 模块 | 优先级 | MVP 设计内容 |
| --- | --- | --- |
| 智能咨询 | P0 | 支持劳动合同自然语言提问，输出结论摘要、法律依据、维权流程、注意事项。 |
| 依据溯源 | P0 | 展示来源名称、条款或章节、文档标题、相关片段，支持展开查看。 |
| 维权流程 | P0 | 针对争议问题输出协商、投诉、仲裁、诉讼等路径及证据清单。 |

| 安全合规 | P0 | 提供免责声明、范围识别、安全拒答、敏感信息提示和后台接口鉴权。 |
| 合同条款审查 | P1 | 用户粘贴劳动合同条款，系统识别明显风险并给出修改建议。 |
| 问答历史 | P1 | 支持保存咨询记录、继续追问、导出结果。 |

### 3.2 暂不纳入范围

1. 不提供律师代理、法律文书代写或付费法律服务。
2. 不承诺覆盖所有地方性劳动政策，首版以全国性法规政策和示例材料为主。
3. 不训练专有大模型，首版通过大模型 API 或本地开源模型加 RAG 实现。
4. 不将用户上传或输入的敏感个人材料作为训练数据。
5. 不对仲裁或诉讼结果作确定性胜诉承诺。

## 6. 业务流程设计

### 6.1 智能问答流程

```text
1. 用户进入咨询页，输入劳动合同相关问题。
2. 前端提交问题、会话 ID、问题类型到后端。
3. 后端进行问题分类和范围识别。
4. 如果问题超出劳动合同范围，返回安全拒答。
5. 如果问题属于劳动合同范围，后端从知识库检索相关片段。
6. 系统按相关度、时效性、来源权威性对片段排序。
7. 后端构造提示词，要求大模型基于检索片段生成回答。
8. 大模型返回结构化回答。
9. 后端校验回答是否包含结论、依据、流程、注意事项。
10. 前端展示答案、依据片段、继续追问、保存和反馈入口。
```

### 6.2 知识库入库流程

```text
1. 管理员上传法规、司法解释、人社政策、FAQ 或示例合同。
2. 系统保存原始文件和文档元数据。
3. 文档解析服务提取文本内容。
4. 系统按条款、章节或自然段进行文本切分。
5. 每个片段绑定文档名、章节、条款号、来源 URL、发布时间、入库时间等元数据。
6. Embedding 服务生成向量。
7. Elasticsearch 写入文本索引和向量索引。
8. MySQL 保存文档、片段、版本和启停用状态。
9. 管理员检查入库结果，确认启用后参与正式问答。
```

### 6.3 反馈优化流程

```text
1. 用户对回答点赞、点踩或提交错误反馈。
2. 系统记录问题、答案、引用片段、反馈类型和反馈内容。
3. 管理员查看低评分问答。
4. 管理员判断问题来源：知识缺失、检索不准、提示词不佳或模型表达问题。
5. 管理员更新知识库、FAQ 或提示词配置。
6. 系统保留处理记录，用于后续评测和复盘。
```

## 7. 功能模块设计

### 7.1 智能咨询模块

#### 7.1.1 支持问题类型

首版覆盖以下 10 类高频劳动合同场景：

1. 合同签订
2. 试用期
3. 工资支付
4. 加班工资
5. 休息休假
6. 社保
7. 解除或终止劳动合同
8. 经济补偿
9. 劳动仲裁
10. 证据准备

#### 7.1.2 回答结构

所有有效回答固定输出以下结构：

```text
1. 结论摘要：先给出简明判断。
2. 法律依据：列出法规、司法解释、政策文件或知识库来源。
3. 维权流程：说明协商、投诉、仲裁、诉讼等路径。
4. 证据清单：提示合同、工资流水、考勤记录、聊天记录等材料。
5. 注意事项：提示时效、地方差异、复杂案件咨询专业律师。
```

#### 7.1.3 安全拒答

以下情况触发安全拒答：

1. 问题明显超出劳动合同或劳动争议范围。
2. 知识库没有足够依据支持回答。
3. 用户要求系统预测仲裁或诉讼“一定能赢”。
4. 用户要求规避法律、伪造证据或恶意维权。
5. 问题涉及高度个案化策略且证据复杂。

安全拒答不直接结束服务，应引导用户补充信息、查看权威渠道或咨询专业机构。

### 7.2 依据溯源模块

答案中的每条关键法律结论至少关联一个来源。来源展示字段包括：

| 字段 | 说明 |
| --- | --- |
| sourceTitle | 来源名称，例如《劳动合同法》。 |
| articleNo | 条款号、章节号或政策标题。 |
| sourceType | 法律法规、司法解释、人社政策、FAQ、示例合同。 |
| publishDate | 发布日期。 |
| effectiveStatus | 有效、待确认、停用。 |
| snippet | 命中的原文摘要或片段。 |
| sourceUrl | 来源链接。 |

排序规则：法律法规优先于司法解释，司法解释优先于人社政策，人社政策优先于 FAQ 和示例材料；同等权威性下按相关度和时效性排序。

### 7.3 维权流程与证据清单模块

争议类回答必须包含行动路径。通用流程如下：

1. 与用人单位沟通并保留沟通记录。
2. 整理劳动关系、工资、考勤、加班、解除通知等证据。
3. 向劳动保障监察部门投诉或举报。
4. 在仲裁时效内申请劳动仲裁。
5. 对仲裁结果不服或符合法定条件时，依法提起诉讼。

常见证据清单：

| 类型 | 示例 |
| --- | --- |
| 劳动关系 | 劳动合同、入职通知、工牌、工作邮件、工作群记录。 |
| 工资支付 | 工资条、银行流水、转账记录、个税记录。 |
| 工时考勤 | 打卡记录、排班表、加班通知、审批记录。 |
| 沟通记录 | 微信聊天、短信、邮件、录音说明。 |
| 工作事实 | 工作成果、客户沟通、系统账号、绩效记录。 |
| 社保记录 | 社保缴纳证明、公积金缴存记录。 |

### 7.4 合同条款审查模块

合同审查为 P1 功能，首版只做辅助风险提示，不提供完整法律审查结论。

| 审查类型 | 识别内容 | 输出内容 |
| --- | --- | --- |
| 试用期 | 试用期过长、试用期工资过低、重复约定试用期。 | 风险等级、依据、建议表述。 |
| 工资工时 | 工资结构不清、加班费约定模糊、休息休假缺失。 | 风险说明、需补充字段。 |
| 解除终止 | 单方解除条件过宽、违约金约定不合理。 | 风险解释、谈判建议。 |
| 保密竞业 | 竞业限制对象、期限、补偿约定不清。 | 风险点、补充建议。 |

风险等级设计：

| 等级 | 说明 |
| --- | --- |
| 高风险 | 明显可能违法或严重损害劳动者权益。 |
| 中风险 | 表述不清或存在争议空间。 |
| 低风险 | 需要补充说明或提醒用户关注。 |

### 7.5 知识库管理模块

管理员功能包括：

1. 上传法规、司法解释、人社政策、FAQ、示例合同。
2. 自动解析 PDF、DOCX、TXT 中至少一种格式；MVP 优先支持 PDF 或 TXT。
3. 自动切分文本并生成向量索引。
4. 编辑标题、来源、发布日、适用范围、版本号等元数据。
5. 启用、停用、删除或标记待确认文档。
6. 查看文档入库状态、切片数量、索引状态和最近更新时间。
7. 维护高频 FAQ，提高常见问题回答稳定性。

### 7.6 问答历史与反馈模块

| 功能 | 设计说明 |
| --- | --- |
| 会话历史 | 登录用户可保存问答历史，游客仅保留当前会话。 |
| 连续追问 | 后端保存会话上下文摘要，不长期保存用户敏感原文。 |
| 答案导出 | 支持导出文本形式的咨询结果和引用来源。 |
| 点赞点踩 | 记录用户对答案质量的评价。 |
| 错误反馈 | 用户可描述错误原因，管理员可标记处理状态。 |

## 10. RAG 设计

### 10.1 检索策略

MVP 建议采用混合检索：

1. 关键词检索：适合条款号、法规名称、专业术语匹配。
2. 向量检索：适合自然语言问题语义匹配。
3. 元数据过滤：过滤停用、过期、待确认、非适用范围的文档。
4. 重排序：按权威性、相关度、时效性综合排序。

### 10.2 知识片段结构

```json
{
  "chunkId": "kc_000001",
  "documentId": "doc_000001",
  "sourceTitle": "中华人民共和国劳动合同法",
  "sourceType": "法律法规",
  "chapter": "第二章 劳动合同的订立",

```

### 10.3 提示词约束

后端构造提示词时必须包含以下约束：

1. 只能基于检索片段回答，不得编造条款。
2. 没有依据时必须说明依据不足。
3. 不得承诺仲裁或诉讼结果。
4. 必须输出结论摘要、法律依据、维权流程、证据清单、注意事项。
5. 必须提示系统不替代律师或仲裁、法院意见。

## 13. 权限设计

| 角色 | 权限 | 限制 |
| --- | --- | --- |
| 游客/普通用户 | 提问、查看答案、查看引用、提交反馈。 | 不能上传知识库文件，不能查看他人会话。 |
| 登录用户 | 保存问答历史、导出结果、继续追问。 | 只能管理自己的记录。 |
| 管理员 | 上传或停用知识文档、查看反馈、维护 FAQ、配置提示词。 | 不应查看用户敏感原文，除非获得授权。 |
| 项目演示评审 | 查看演示账号、测试问答、查看系统说明。 | 只读访问，不修改知识库。 |

MVP 实现建议：先完成游客问答与管理员知识库维护；如实训周期紧张，账号权限可使用简单登录和角色字段实现，不引入复杂权限框架。

## 14. 安全与合规设计

### 14.1 免责声明

在首页、答案页和导出结果中展示：

```text
本系统仅提供劳动合同法律信息检索与咨询辅助，不构成正式法律意见，不替代律师、仲裁机构或法院意见。
```

### 14.2 隐私保护

1. 前端提示用户不要输入身份证号、手机号、家庭住址等敏感信息。
2. 后端日志不明文记录身份证号、手机号、银行卡号等敏感信息。
3. 用户个人材料仅用于当前会话分析，不进入公共知识库。
4. 管理员默认只能查看反馈摘要，不直接查看用户敏感原文。

### 14.3 大模型安全

1. 检索不到依据时拒绝确定性回答。
2. 禁止模型编造法律条款、文号和流程。
3. 对超出范围的问题进行范围识别和安全拒答。
4. 对“预测必胜”“规避法律”“伪造证据”等问题拒答并提示合法路径。

## 19. 当前设计假设

1. 首版优先做全国性法规、司法解释和人社政策，不做全量地方政策覆盖。
2. 用户权限在 MVP 阶段可以简化实现，但管理员知识库维护接口必须具备基础鉴权。
3. 文档存储首版可使用本地文件系统，后续可切换为对象存储。
4. Elasticsearch 同时承担关键词检索和向量检索；如果部署条件有限，可先用内存向量库或轻量级向量数据库替代。
5. 大模型 API 使用 Spring AI OpenAI Starter 封装；具体模型由环境变量或配置文件指定。
6. 法律依据的最新有效性由管理员入库前确认，系统负责记录来源、发布日期和版本状态。
